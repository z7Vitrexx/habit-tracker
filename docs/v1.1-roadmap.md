# Habit Tracker Pro - v1.1 Roadmap

## 1. Einleitung

### Einordnung
- **v1.0** ist funktional releasebereit und bietet alle Kernfunktionen für private Habit-Tracking
- **v1.1** fokussiert sich auf Alltagstauglichkeit, Klarheit, Performance und Produktreife
- Diese Roadmap ist eine interne Produkt-/Release-Planung für die Weiterentwicklung

### Zielsetzung
Verbesserung des bestehenden Produkts statt neuer Großfeatures. Qualität vor Funktionsmenge.

---

## 2. Version 1.1 – Höchste Priorität (Must-Have)

### Reminder-UX verbessern
- **Übersicht aktiver Erinnerungen:** Zentrale Ansicht aller eingerichteten Erinnerungen
- **Klarere Anzeige der nächsten Erinnerung:** Wann genau kommt die nächste Benachrichtigung?
- **Bessere Kommunikation der lokalen Browser-Grenzen:** Ehrliche UX-Hinweise prominent platzieren
- **Optional Snooze / später erinnern:** Falls lokal sauber machbar, ohne systemweite Erinnerungen

### Statistik verständlicher machen
- **Bessere Zusammenfassungen pro Habit:** Klarere KPIs und Trends auf einen Blick
- **Klarere Wochenübersicht:** Visuelle Woche mit Fortschritt und Ausfallquoten
- **Bessere Darstellung quantitativer Habits:** Charts und Trends für numerische Werte
- **Saubere Trennung zwischen verpasst, übersprungen und nicht geplant:** Visuell klar unterscheiden

### Verlauf nützlicher machen
- **Schnellfilter:** Nur verpasste / nur übersprungene / nur mit Notiz
- **Bessere Tag-Details:** Mehr Kontext und schnellere Bearbeitung
- **Bessere Orientierung in Listen:** Scroll-Positionen, Datumsmarken, Zusammenfassungen
- **Mehr direkte Bearbeitung:** Inline-Edit wo sinnvoll, weniger Dialoge

### Performance verbessern
- **Bundle-Größe reduzieren:** Aktuell ~960KB, Ziel < 700KB
- **Unnötige Komponenten/Libraries prüfen:** Code-Splitting, Tree-Shaking optimieren
- **Charts/Statistik bei Bedarf lazy laden:** Nur laden wenn Statistik-Tab aktiv
- **Unnötige Imports und Dev-Reste entfernen:** Final cleanup

## 6. Release-Logik

### Fokus für v1.1
- **Kein Scope-Explosion:** v1.1 soll kein Monster-Release werden
- **Verbesserung statt Expansion:** Bestehendes Produkt polieren
- **Qualität vor Quantität:** Weniger Features, aber besser gemacht
- **Alltagstauglichkeit:** App soll sich im täglichen Gebrauch besser anfühlen

### Release-Strategie
- **Iterativ:** Kleine, häufige Updates statt großer Releases
- **Feedback-getrieben:** Nutzerfeedback aus v1.0 einarbeiten
- **Stabilität:** Jede Version muss stabiler als die vorherige sein

## 7. Technische Überlegungen

### Performance-Ziele
- **Bundle-Größe:** < 700KB (aktuell ~960KB)
- **Ladezeit:** < 2s auf 3G
- **Memory:** < 50MB im normalen Gebrauch
- **Offline:** Volle Funktionalität ohne Internet

### Code-Qualität
- **TypeScript:** 100% typisiert
- **Tests:** Unit-Tests für Kernlogik
- **Lint:** 0 Fehler, 0 Warnungen
- **Bundle-Analyse:** Regelmäßige Überprüfung der Bundle-Größe

### Architektur
- **Component-Splitting:** Lazy loading wo sinnvoll
- **State-Management:** Optimierung der React-Performance
- **Database:** IndexedDB-Optimierungen
- **PWA:** Service Worker und Caching verbessern
